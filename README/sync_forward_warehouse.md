

1    Сразу вывести в “Рейсах” актуальную складскую информацию по контейнеру (мест/вес)

        Источник: агрегат warehouse_item_out (status='sended') + вес из warehouse_item_stock.

        Это уже логически правильно ложится в текущую колонку “Склад (мест/вес)” в таблице контейнеров рейса.

        Сейчас в рейсах контейнеры уже берутся из *_flight_list_containers с приоритетом над snapshot полями рейса — значит это место идеально для “живого” вывода.

2    Убрать концепцию warehouse_packages_json как обязательного слоя

        Не хранить лишний warehouse JSON в контейнерной таблице, если нам нужен always-live view.

        Warehouse остаётся “source of truth” в warehouse_item_out.

        Снимет риск рассинхрона “в json одно, в out другое”.

        (Сейчас это поле у нас добавлено — его можно оставить временно как legacy, но не использовать в расчётах).
готово
3    Оставить только forwarder_packages_json как snapshot внешнего состояния

        Здесь ты полностью прав: форвард — внешний источник, его snapshot полезно хранить локально.

        Обновление snapshot делать:

            по ручной кнопке sync в рейсах,

            и автоматически после успешного добавления посылки в контейнер форварда (run_add_package_to_container.php + список из run_list_container.php).

        Это даёт “факт от форварда” + “факт от склада” для сверки.

4    Сверку считать на чтении

        warehouse_count/weight — live агрегат из warehouse.

        forwarder_count/weight — из forwarder_packages_json (или из прямого sync результата).

        На UI показывать:

            matched / mismatch,

            forwarder_packages_synced_at,

            (опционально) warehouse_aggregated_at как время расчёта.

5    Автосинхронизация при confirm send

        При подтверждении отправки в warehouse_item_out_confirm_send:

            обновили warehouse_item_out,

            зарегистрировали посылку у форварда,

            пересняли forwarder_packages_json,

            обновили compare-статус.

        Это соответствует твоему бизнес-процессу “каждое действие с посылкой -> синхра с форвардом”.

6    Оптимизация производительности (если понадобится)

        Если live-агрегаты станут тяжёлыми, делаем materialized-агрегат (таблица/кэш), но это уже этап 2.

        Сначала запускаем без лишних кэшей.
