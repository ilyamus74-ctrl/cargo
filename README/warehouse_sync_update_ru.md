# Обновление синхронизации с форвардом (warehouse sync)

Обновили ручную синхронизацию по кнопке **sync**: теперь регистрация посылки выполняется через `run_add_package.php`. Логика стала более предсказуемой для операторов и проще в диагностике спорных случаев.

Ключевое правило по клиенту и статусу теперь фиксированное и прозрачное:
- `client-id = 0` → `status-id = 36`
- `client-id > 0` → `status-id = 37`

Также изменили валидацию: если клиент выбран (`client_id > 0`), отсутствие `receiver_name` больше **не** блокирует отправку в sync. Это убирает лишние отказы в кейсах, где клиент уже определён корректно.

После sync в интерфейсе показываем ключевые поля ответа форварда: `status`, `http_status`, `submit_case`, `internal_id`, `status_id_effective`. Полный технический ответ (`forwarder_response`) по-прежнему доступен в консоли.

Отдельно важно: статус больше не форсится в `half_sync`. При успешной регистрации фиксируется конкретный `success`, чтобы итог операции был однозначным и понятным.

Эти изменения нужны для трёх практических вещей: меньше ложных блокировок, корректнее применение правил статусов и быстрее разбор проблемных регистраций.

## Что изменилось

- sync-регистрация выполняется через `run_add_package.php`.
- Введено правило соответствия `client-id/status-id`: `0 → 36`, `>0 → 37`.
- Для `client_id > 0` отсутствие `receiver_name` не блокирует отправку.
- В UI после sync выводятся поля: `status`, `http_status`, `submit_case`, `internal_id`, `status_id_effective`.
- Полный `forwarder_response` выводится в консоль.
- Убрано принудительное `half_sync`; при успехе фиксируется `success`.

## Что делать оператору

1. После sync проверить alert с полями ответа форварда.
2. Если кейс спорный, открыть консоль и скопировать `forwarder_response` для передачи в разбор.
3. Оценивать результат по фактическим полям ответа, а не по промежуточным предположениям о статусе.

Короткий пример ответа:
- `status: success`
- `http_status: 200`
- `submit_case: created`
