# План внедрения: ранняя регистрация посылки у форварда после `commit_item_in_batch`

## 1) Цель

Сделать двухэтапную модель синхронизации:

1. **Предрегистрация у форварда** сразу после приемки на склад (после переноса из `warehouse_item_in` в `warehouse_item_stock`) через `run_add_package.php`.
2. **Фактическая отгрузка** в контейнер рейса позже, на этапе `warehouse_item_out_confirm_send`, через `run_add_package_to_container.php`.

Это гарантирует, что форвард заранее знает о входящей посылке, но факт физической отправки фиксируется отдельно.

---

## 2) Текущее состояние (как есть)

- `commit_item_in_batch`:
  - переносит batch из `warehouse_item_in` в `warehouse_item_stock`;
  - ставит `committed = 1`;
  - не выполняет регистрацию посылки у форварда.
- `warehouse_item_out_confirm_send`:
  - переводит `warehouse_item_out.status` в `sended`;
  - вызывает `run_add_package_to_container.php`;
  - обновляет snapshot контейнера у форварда.
- `warehouse_sync`/`reconcile`:
  - подтягивает статусы из report-таблиц форварда;
  - переводит статусы в `warehouse_item_out` по map-логике.

## 3) Целевой процесс (to-be)

1. Сканер/оператор заполняет `warehouse_item_in`.
2. Нажатие **"Завершить партию (на склад)"** переносит данные в `warehouse_item_stock`.
3. Для новых записей запускается **автоматическая предрегистрация** через `run_add_package.php`.
4. При успешной регистрации у форварда:
   - запись **не** создается в `warehouse_item_out` на этом шаге;
   - пишется audit + технический trace по результату регистрации;
   - при необходимости обновляются только поля/метки в `warehouse_item_stock` (или отдельном registration-log).
5. Если поля обязательные для форварда не заполнены:
   - регистрация не выполняется;
   - у посылки ставится понятная причина блокировки (status/message/audit).
6. Позже `run_report.php`/reconcile, на основании статусов из репорта, создает/обновляет запись в `warehouse_item_out` и выставляет `confirmed_sync`/`to_send`/`sended` по map-логике.
7. На `confirm_send` выполняется только контейнеризация (`run_add_package_to_container.php`) и фиксация `sended`.

## 4) Обязательные поля для предрегистрации

### 4.1 Глобально обязательные

- `tracking_no` (или fallback `tuid`)
- `receiver_country_code`
- `receiver_company` (форвард)
- `weight_kg`
- `receiver_name`
- `receiver_address`

### 4.2 Бизнес-обязательные (из вашего процесса)

- ДопИнфо / addons (`addons_json`)
- `tariff_type`
- `category`
- ячейка клиента
- получатель
- страна получателя

> Примечание: финальный источник каждого поля должен быть закреплен в коде (из `warehouse_item_stock` и/или `addons_json`, с fallback на правила коннектора).

### 4.4 Уточнение по текущей репорт-операции

В существующей операции «Репорт» (настройка `connector_report_*`) физические параметры посылки приходят ограниченно; по сути в стандартном выгружаемом наборе используется в т.ч. `weight`, а ключевые поля для предрегистрации должны задаваться отдельно и явно в нашей модели.

Вывод для реализации:

- не полагаться на репорт как на источник обязательных полей для `run_add_package.php`;
- обязательность и источник полей предрегистрации держать в отдельной конфигурации (см. раздел 6.3).

### 4.3 Валидация

- Если хотя бы одно поле отсутствует: не отправлять в `run_add_package.php`.
- Ошибки валидатора должны быть агрегированы и записаны в человекочитаемом виде.

## 10) Тест-кейсы (обязательные)

## 10.1 Позитивные

- Полный пакет обязательных полей -> `run_add_package.php` возвращает success -> локальный статус обновлен.
- Повторный запуск для того же трека -> идемпотентный success.

## 10.2 Негативные

- Нет `tariff_type` / `category` / клиента / страны -> регистрация не уходит, понятная ошибка в ответе.
- Неактивный коннектор -> controlled error.
- Таймаут/500 у форварда -> ошибка логируется, commit batch не ломается полностью.

## 10.3 E2E

- Приемка -> commit -> предрегистрация у форварда -> report -> `to_send` -> confirm_send -> container add -> `sended`.

---

## 11) Критерии приемки

1. После commit batch форвард получает предрегистрацию по валидным посылкам.
2. Невалидные посылки не отправляются и явно видны оператору.
3. Дубли не создаются.
4. Ошибки интеграции прозрачны в истории/audit.
5. Этап `confirm_send` остается этапом контейнеризации и факта отправки, а не первичной регистрации.

---

## 12) Риски и решения

- **Риск:** форвард меняет формат формы `/collector/add`.
  - **Решение:** использовать существующий адаптивный парсинг формы в `run_add_package.php` + мониторинг ошибок.
- **Риск:** рост времени `commit_item_in_batch`.
  - **Решение:** вынести предрегистрацию в очередь (этап 2), сначала синхронно с ограничением размера batch.
- **Риск:** неполные данные из сканера.
  - **Решение:** жесткий валидатор + UX-подсветка обязательных полей до нажатия commit.

---

## 13) Что делаем первым (конкретный старт)

1. Утверждаем список обязательных полей и источник каждого поля.
2. Внедряем в `commit_item_in_batch` dry-run валидацию без отправки во форвард (1 итерация).
3. После стабилизации включаем реальный вызов `run_add_package.php` флагом (`WAREHOUSE_FORWARDER_PRE_REGISTER=1`).
