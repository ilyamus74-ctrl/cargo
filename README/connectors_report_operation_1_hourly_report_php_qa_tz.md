# ТЗ для QA: проверка запуска `connectors_report_operation_1_hourly` через `report_php`

## 1) Цель
Подтвердить, что периодическая задача `connectors_report_operation_1_hourly` запускается и исполняется именно через PHP-операцию `report_php`, а не через Node/legacy report-механизм.

## 2) Область проверки
В рамках проверки QA подтверждает:
- корректную маршрутизацию запуска к операции `operation_id=report_php`;
- отсутствие фактического исполнения legacy/node-пути для данного прогона;
- корректный результат импорта (статус выполнения, сообщение, количество импортированных строк);
- консистентность данных в БД до/после выполнения.

Вне рамок:
- функциональное покрытие всех коннекторов;
- нагрузочное тестирование и долгосрочный мониторинг;
- рефакторинг scheduler/worker логики.

## 3) Предусловия
Перед стартом теста должны выполняться условия:
1. В тестовой среде доступен коннектор `connector_id=13`.
2. Для коннектора существует и доступна операция `report_php`.
3. Планировщик/раннер задач активен (или есть возможность ручного запуска операции).
4. QA имеет доступ:
   - к UI карточки/журнала запусков коннектора;
   - к таблицам БД для проверки артефактов импорта;
   - к логам выполнения (если применимо).
5. На период теста отключены внешние факторы, искажающие результат (параллельные ручные прогоны того же набора данных).

## 4) Тестовые данные
### Базовый набор
- `connector_id=13`
- целевая операция: `operation_id=report_php`
- целевая задача: `connectors_report_operation_1_hourly`

### Контрольный payload для точечного прогона
Используется принудительный `payload_json` для валидации маршрутизации:

```json
{
  "connector_id": 13,
  "operation_id": "report_php"
}
```

Назначение: исключить двусмысленность выбора операции и явно зафиксировать маршрут выполнения на `report_php`.

## 5) Позитивные сценарии
### Сценарий P1: плановый/ручной запуск hourly-задачи
**Шаги:**
1. Запустить `connectors_report_operation_1_hourly` штатным способом (scheduler или ручной триггер).
2. Дождаться завершения выполнения.
3. Проверить в журнале запусков/операций:
   - выбран `operation_id=report_php`;
   - выполнение завершилось со статусом успеха.
4. Сверить артефакты импорта в UI и БД.

**Ожидаемый результат:**
- задача отработала через `report_php`;
- статус выполнения успешный;
- импорт выполнен, значения `imported_rows` и служебные поля консистентны.

### Сценарий P2: принудительный запуск с `payload_json`
**Шаги:**
1. Инициировать точечный прогон с payload:
   - `connector_id=13`
   - `operation_id=report_php`
2. Дождаться завершения.
3. Проверить фактический маршрут выполнения и результат импорта.

**Ожидаемый результат:**
- раннер использует именно `report_php`;
- отсутствуют признаки fallback в node/legacy report;
- результат импорта валиден по статусу/сообщению/количеству строк.

## 6) Негативные сценарии
### Сценарий N1: невалидный `operation_id`
**Шаги:**
1. Выполнить запуск с `payload_json`, где `operation_id` не существует для коннектора.
2. Проверить обработку ошибки.

**Ожидаемый результат:**
- запуск завершается контролируемой ошибкой;
- ошибка диагностируема (понятный `status/message`);
- отсутствует «тихий» fallback на node/legacy маршрут.

### Сценарий N2: некорректный `connector_id`
**Шаги:**
1. Запустить задачу с несуществующим/недоступным `connector_id`.
2. Проверить итоговый статус и диагностическое сообщение.

**Ожидаемый результат:**
- корректное отклонение запуска;
- фиксируемый текст ошибки;
- отсутствие частичного импорта.

### Сценарий N3: пустой или битый payload
**Шаги:**
1. Инициировать запуск с пустым/некорректным `payload_json`.
2. Проверить валидацию входных данных и итоговую запись в журнале.

**Ожидаемый результат:**
- запуск не приводит к неконтролируемому выполнению;
- ошибка явно отражена в `status/message`;
- данные в целевых таблицах не изменены.

## 7) Критерии приемки
Тест считается пройденным, если одновременно выполнены все условия:
1. Для `connectors_report_operation_1_hourly` подтвержден маршрут запуска через `operation_id=report_php`.
2. Не обнаружены признаки выполнения через node/legacy report для проверяемого прогона.
3. Позитивные сценарии P1 и P2 завершены успешно.
4. Негативные сценарии N1–N3 завершаются предсказуемо и с корректной диагностикой.
5. Полный комплект артефактов приложен к QA-отчету.

## 8) Риски
- Параллельные запуски могут исказить сопоставление `run_id` и фактического результата.
- Кэш/ретраи в очереди могут создать ложное впечатление повторного выполнения.
- Неполные права QA к БД/логам усложняют доказательство маршрутизации.
- Внешняя нестабильность источника данных может повлиять на `imported_rows`, не являясь дефектом маршрута.

## 9) Rollback
Если проверка выявила запуск не через `report_php` или нестабильное поведение:
1. Остановить дальнейшие релизы/раскатку изменения маршрутизации.
2. Зафиксировать проблемный `run_id` и артефакты.
3. Вернуть предыдущую стабильную конфигурацию запуска (до изменения маршрута), либо временно отключить проблемный триггер.
4. Повторно провести smoke-проверку на подтверждение восстановления стабильного поведения.

## 10) Обязательные артефакты в QA-отчете
QA обязан приложить:
1. `run_id` проверяемого запуска (для каждого сценария).
2. Финальные `status` и `message`.
3. Значение `imported_rows`.
4. UI-скриншоты:
   - карточка/журнал запуска;
   - детали выполнения операции.
5. SQL-выгрузки до/после выполнения:
   - запросы и результаты, подтверждающие изменения (или их отсутствие для негативных кейсов).

Рекомендуемый минимум для трассируемости:
- отметка времени запуска;
- инициатор запуска (scheduler/manual);
- использованный payload (в т.ч. принудительный `operation_id=report_php` для P2).
