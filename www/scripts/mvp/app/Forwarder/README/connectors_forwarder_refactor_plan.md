# План рефакторинга Connectors/Forwarder (цель + шаги)

> Дата: 2026-04-16  
> Статус: Draft v1 (для обсуждения и правок)

## 1) Цель работ

Сделать систему коннекторов **масштабируемой под разные forward-системы** (COLLI, CAMEX, и др.), но при этом **простой для настройки обычным пользователем**.

### Что должно получиться на выходе

1. Единое ядро выполнения операций (`ForwarderCore`), без копирования общей логики в каждую систему.
2. Системно-специфичные адаптеры (`Connectors/COLLI`, `Connectors/CAMEX`) с минимальным набором отличий.
3. Двухуровневая настройка операций:
   - **Simple mode** (короткая форма для пользователя);
   - **Advanced mode** (полный JSON/DSL для инженера).
4. Убрать из пользовательского слоя лишние детали:
   - логины/пароли,
   - `script_path`, длинные `args`,
   - ручной `target_table` по умолчанию.
5. Ввести единый каталог операций (presets) и понятные обязательные поля.

---

## 2) Контекст и текущие проблемы

1. Настройки операций сейчас перегружены деталями runtime (headers/csrf/script_path/args), из-за чего ими тяжело пользоваться без технической помощи.
2. Операции тесно связаны с конкретными скриптами (`run_*.php`) и путями в файловой системе.
3. Для новых систем есть риск пойти в копии папок `Forwarder`, что ускоряет старт, но ухудшает поддержку.
4. Пользовательская задача (например, «обновить отчет») смешана с техническими деталями выполнения.

---

## 3) Принципы будущего решения

1. **Core first**: общие HTTP/session/retry/logging/DTO — только в ядре.
2. **Adapter per system**: отличия конкретных систем живут в адаптерах.
3. **Operation as product**: операция описывает бизнес-цель, а не shell-команду.
4. **Secure by default**: креды не хранятся в operation JSON.
5. **Backward-compatible migration**: на переходный период оставляем поддержку старого формата.

---

## 4) Целевая архитектура (высокоуровнево)

```text
www/scripts/mvp/app/
  ForwarderCore/
    Http/
    Session/
    DTO/
    Logging/
    Orchestrator/
    Contracts/
      ConnectorAdapterInterface.php

  Connectors/
    COLLI/
      ColliAdapter.php
      ColliOperationMap.php
      ColliFieldMap.php
    CAMEX/
      CamexAdapter.php
      CamexOperationMap.php
      CamexFieldMap.php

  runners/
    run_operation.php
```

### Смысл
- `ForwarderCore` ничего не знает о конкретном форварде.
- Адаптер знает endpoint’ы, особенности auth и mapping полей.
- Раннер получает `connector_id + operation_code` и резолвит адаптер/handler.

---

## 5) Модель операций (как упрощаем)

## 5.1 Simple mode (для пользователя)
Минимальные поля:
- Тип операции (`report_sync`, `flight_list_sync`, `add_flight`, ...)
- Коннектор
- Период (если нужен)
- Включено/выключено
- (опц.) Таблица назначения override

## 5.2 Advanced mode (для инженера)
Полный JSON/DSL с деталями transport, headers, сценариями и т.д.

## 5.3 Resolver слой (главная идея)
Вместо хранения `script_path` в пользовательской операции:
- храним `adapter_code + operation_code + kind`;
- Resolver внутри backend выбирает конкретный handler/скрипт.

---

## 6) Таблицы и данные

### 6.1 target_table
- По умолчанию вычисляется автоматически по шаблону:
  - `connector_{system}_{country}_{operation}`
- Пользователь может переопределить только при необходимости (advanced).

### 6.2 credentials
- Логины/пароли — только в карточке коннектора / vault / env.
- Operation JSON не содержит секретов.

### 6.3 field_mapping
- Для каждой операции вводим:
  - `required_fields`
  - `optional_fields`
  - `aliases`
- UI предупреждает, если обязательные поля не покрыты.

## 7) Этапы реализации

## Этап 0. Подготовка (1–2 дня)
- Зафиксировать список operation_code и их назначение.
- Определить обязательные поля для ключевых операций:
  - report sync,
  - flight list sync,
  - add/edit/delete flight,
  - add/delete container.

## Этап 1. Контракт адаптера (2–3 дня)
- Описать `ConnectorAdapterInterface`.
- Зафиксировать единый формат `OperationResult`.
- Определить обязательные методы по операциям.

## Этап 2. Operation Resolver (2–4 дня)
- Реализовать резолвинг `adapter_code + operation_code + kind -> handler`.
- Сохранить fallback на legacy `script_path`.

## Этап 3. Simple/Advanced режимы (3–5 дней)
- Добавить API/форму для simple-конфига.
- Генерировать advanced-конфиг автоматически.
- Добавить валидацию required/optional fields.

## Этап 4. Пилот: COLLI (3–5 дней)
- Перенести 2 операции:
  - report download+import,
  - flight list sync.
- Проверить совместимость и логи.

## Этап 5. CAMEX + обобщение (3–5 дней)
- Добавить второй адаптер.
- Дообобщить Core при выявленных расхождениях.

## Этап 6. Cleanup (2–3 дня)
- Постепенно убрать прямые `script_path` из UI.
- Оставить legacy только как технический fallback.

---

## 8) Критерии успеха

1. Пользователь может создать и запустить ключевую операцию без редактирования raw JSON.
2. Добавление новой системы не требует копии всей папки `Forwarder`.
3. Секреты отсутствуют в operation payload.
4. Время диагностики ошибок сокращается (понятные статусы и логи).
5. Старые операции продолжают работать в период миграции.

---

## 9) Риски и как снижаем

1. **Риск:** ломаем текущие рабочие операции.
   - **Снижение:** fallback на legacy-конфиг + поэтапная миграция.
2. **Риск:** Core станет слишком абстрактным.
   - **Снижение:** сначала вынести только реально общий код, без overengineering.
3. **Риск:** пользователи всё равно уходят в advanced.
   - **Снижение:** качественные presets и валидация в simple mode.
