# План внедрения: шаблон лейбла форварда (коннектор)

## Контекст и договоренности

- Кнопка **"Шаблон лейбла"** добавляется в списке коннекторов **между колонками/кнопками "Операции" и "Действия"**.
- Используем **существующую модалку коннектора** (fullscreenModal), без отдельного нового окна.
- В редакторе делаем **минимальную валидацию** шаблона.
- Добавляем кнопку **"Тест печати шаблона"**.
- Отдельное версионирование шаблонов **не внедряем**.
- Изменения шаблона логируем через `audit_log` (кто/когда/что менял).

## 1) UI/UX изменения

### 1.1 Список коннекторов

**Файл:** `templates/cells_NA_API_connectors.html`

- Добавить новую колонку `Шаблон` между `Операции` и `Действия`.
- Для каждой строки коннектора добавить кнопку:
  - Текст: `Шаблон лейбла`
  - Класс: `btn btn-sm btn-outline-warning`
  - Действие: `data-core-action="form_connector_label_template"`
  - Параметр: `data-connector-id="..."`

### 1.2 Контент в существующей модалке

**Файл:** новый шаблон, например `templates/cells_NA_API_connector_label_template_modal.html`

В `fullscreenModal` рендерится split-view:

- Левая колонка:
  - `template_code` (readonly/селект)

  - краткая памятка по доступным плейсхолдерам
  - блок валидации
- Правая колонка:
  - предпросмотр HTML

Кнопки внизу формы:
- `Сохранить шаблон`
- `Проверить шаблон` (валидация без сохранения)
- `Тест печати шаблона`

## 2) Backend действия (warehouse_sync_actions.php)

**Файл:** `www/api/warehouse/warehouse_sync_actions.php`

Добавить новые `action`:

1. `form_connector_label_template`
   - Вход: `connector_id`
   - Выход: HTML формы для модалки
   - Источник данных: таблица `forwarder_label_templates` (активная запись)

2. `save_connector_label_template`
   - Вход: `connector_id`, `template_code`, `template_body`
   - Поведение:
     - минимальная валидация
     - upsert активного шаблона для коннектора
     - `audit_log(..., 'CONNECTOR_LABEL_TEMPLATE_UPDATED', ...)`
   - Выход: JSON `status=ok/error`

3. `validate_connector_label_template`
   - Вход: `connector_id`, `template_body`
   - Проверки:
     - не пустой шаблон
     - базовая проверка плейсхолдеров `{{...}}` (парность/валидные имена)
     - ограничение размера тела (например, до 100KB)
   - Выход: `warnings[]`, `errors[]`, `status`

4. `test_print_connector_label_template`
   - Вход: `connector_id`, `template_body`, опционально `test_track`, `print_device_key`
   - Поведение:
     - получить тестовые данные (по `test_track` или фикстура)
     - прогнать генерацию лейбла тем же кодом, что боевой путь
     - при наличии print устройства + токена отправить тестовую печать
     - иначе вернуть предпросмотр/label_base64
   - Логировать результат в `audit_log(..., 'CONNECTOR_LABEL_TEMPLATE_TEST_PRINT', ...)`

---

## 3) Минимальная валидация (MVP)

Обязательные проверки перед сохранением:

1. `template_body` не пуст.
2. Размер шаблона в пределах лимита.
3. Нет явных битых плейсхолдеров:
   - запрещаем одиночные `{`/`}` для синтаксиса плейсхолдеров;
   - проверяем, что `{{field_name}}` использует только `[a-zA-Z0-9_\.-]`.
4. Рекомендуемая (warning) проверка: присутствует хотя бы один ключевой маркер (например `track`).

Ошибки блокируют сохранение, warning — нет.

---

## 4) Аудит изменений (обязательно)

На события:
- сохранение шаблона;
- тест печати;
- ошибка теста.

Пишем в `audit_log`:
- `connector_id`
- `user_id`
- `template_code`
- hash тела шаблона (например `sha256`), чтобы не хранить весь body в логе
- результат/сообщение об ошибке

## 6) Критерии приемки

1. В таблице коннекторов есть кнопка `Шаблон лейбла` между `Операции` и `Действия`.
2. Нажатие открывает контент в существующей fullscreen-модалке.
3. Шаблон можно валидировать и сохранить.
4. Есть кнопка `Тест печати шаблона`; тест отрабатывает и показывает понятный результат.
5. Каждое сохранение и тест фиксируются в `audit_log`.
6. Боевой путь печати при переносе в контейнер не ломается.
